## 1. Executive Summary

### 1.1 Overview

This PRD defines the requirements for implementing real-time streaming of AI-generated summaries from the DashScope API to the frontend. Currently, the system waits for the complete AI response before displaying results. This enhancement will enable users to see the summary being generated in real-time, significantly improving perceived performance and user engagement.

### 1.2 Goals

- **Primary Goal:** Stream AI-generated text chunks to the frontend as they are generated by DashScope API
- **User Experience:** Reduce perceived wait time by showing progress in real-time
- **Technical Goal:** Implement efficient bidirectional streaming (DashScope → Backend → Frontend)
- **Performance Goal:** Maintain or improve system performance while adding streaming capability

### 1.3 Success Criteria

- ✅ Users see summary text appearing in real-time as it's generated
- ✅ Streaming works reliably for summaries of all lengths (short to very long)
- ✅ System gracefully handles streaming failures with fallback to non-streaming mode
- ✅ No performance degradation compared to current non-streaming implementation
- ✅ Streaming can be toggled via configuration without code changes
- ✅ Frontend displays streaming text with proper markdown rendering
- ✅ Connection recovery works if SSE connection is lost during streaming

## 3. Technical Requirements

### 3.1 Backend Requirements

#### 3.1.1 DashScope API Streaming

**Stream Format:**
- DashScope returns Server-Sent Events (SSE) when `stream: true`
- Format: `data: {"output": {...}, "request_id": "..."}`
- End marker: `[DONE]` or empty line
- Content path: `output.choices[0].message.content` (delta or full content)

**Stream Parsing:**
- Parse SSE stream line by line
- Extract content deltas from each chunk
- Accumulate full response for storage
- Handle `enable_thinking` mode if applicable (filter thinking tokens)
- Track token usage from stream metadata

#### 3.1.2 Backend-to-Frontend Streaming

**SSE Event Format:**
```typescript
interface StreamingChunk {
  status: 'generating';
  progress: number; // 85-99%
  chunk: string; // Text delta
  accumulated?: string; // Optional: full accumulated text so far
  message?: string; // Status message
}
```

**Integration Points:**
- Modify `ai.service.ts` to handle streaming responses
- Update `summary.service.ts` to forward chunks via job progress callback
- Use existing `broadcastJobProgress()` function for SSE broadcasting

#### 3.1.3 Error Handling

- **Stream Parsing Errors:** Log error, fallback to non-streaming mode
- **Connection Errors:** Retry with exponential backoff, then fallback
- **Client Disconnection:** Continue processing, accumulate full response
- **Timeout Handling:** Set reasonable timeout for stream processing

### 3.2 Frontend Requirements

#### 3.2.1 Chunk Accumulation

**State Management:**
- Accumulate streaming chunks in `useSummaryStream` hook
- Update `streamedText` state incrementally as chunks arrive
- Handle chunk deduplication (if DashScope sends overlapping content)

**Display Logic:**
- Render markdown in real-time as chunks arrive
- Show typing cursor during active streaming
- Remove cursor when streaming completes
- Debounce markdown parsing for performance (every 100ms)

#### 3.2.2 User Experience

**Visual Feedback:**
- Show "Generating..." status during streaming phase
- Display progress percentage (85% → 99%)
- Animate text appearance smoothly
- Handle rapid chunk arrivals gracefully

**Connection Management:**
- Auto-reconnect if SSE connection drops
- Show connection status indicator
- Handle reconnection during active streaming

### 3.3 Configuration

**Backend Configuration (`config.yaml`):**
```yaml
ai_models:
  stream: true  # Enable/disable streaming
  stream_chunk_delay_ms: 50  # Optional: delay between chunks (throttling)
  stream_fallback_enabled: true  # Fallback to non-streaming on error
```

**Environment Variables:**
- No new environment variables required
- Use existing `DASHSCOPE_API_KEY` and configuration system

interface UseSummaryStreamReturn {

  streamedText: string; // Accumulated streaming text

// In eventSource.onmessage handler:

  setProgress(data.progress);
  setMessage(data.message || null);

## 5. Error Handling & Edge Cases

### 5.1 Backend Error Scenarios

#### 5.1.1 Stream Parsing Failures

**Scenario:** DashScope returns malformed SSE stream

**Handling:**
1. Log error with full context
2. Attempt to parse what's available
3. If critical failure: Fallback to non-streaming mode
4. Notify frontend via error event
5. Continue processing with non-streaming request

**Error Event:**
```json
{
  "status": "error",
  "error": "Stream parsing failed. Using non-streaming mode.",
  "recoverable": true
}
```

#### 5.1.2 Connection Timeouts

**Scenario:** DashScope stream times out or disconnects

**Handling:**
1. Set reasonable timeout (e.g., 5 minutes for stream)
2. On timeout: Fallback to non-streaming mode
3. Log timeout event
4. Notify frontend

#### 5.1.3 Client Disconnection During Streaming

**Scenario:** Frontend SSE connection drops while streaming

**Handling:**
1. Backend continues processing
2. Accumulate full response
3. Store complete summary when done
4. Client can reconnect and receive final result
5. Don't abort AI request

### 5.2 Frontend Error Scenarios

#### 5.2.1 SSE Connection Loss

**Scenario:** Network drops during streaming

**Handling:**
1. Detect connection loss (`eventSource.onerror`)
2. Show reconnection indicator
3. Auto-reconnect with exponential backoff
4. Resume streaming from last received chunk (if supported)
5. If reconnection fails: Show error, allow manual retry

#### 5.2.2 Chunk Ordering Issues

**Scenario:** Chunks arrive out of order (unlikely with SSE)

**Handling:**
1. Include sequence numbers in chunks (if needed)
2. Buffer out-of-order chunks
3. Reorder before displaying
4. Log warning if reordering occurs

#### 5.2.3 Rapid Chunk Arrival

**Scenario:** Many chunks arrive very quickly

**Handling:**
1. Debounce markdown parsing
2. Batch state updates
3. Use `requestAnimationFrame` for smooth rendering
4. Throttle scroll updates

### 5.3 Edge Cases

#### 5.3.1 Empty Chunks

**Handling:** Ignore empty chunks, don't update state

#### 5.3.2 Duplicate Content

**Handling:** Compare new chunk with accumulated text, skip duplicates

#### 5.3.3 Very Long Summaries

**Handling:** 
- Virtual scrolling for display
- Pagination for markdown parsing (if needed)
- Memory management for accumulated text

#### 5.3.4 Streaming Disabled in Config

**Handling:** 
- System works normally without streaming
- No code changes needed
- Frontend receives complete summary on completion

## 6. Performance Considerations

### 6.1 Backend Performance

**Memory Management:**
- Stream chunks should not be buffered unnecessarily
- Process chunks as they arrive
- Accumulate only final content for storage
- Limit in-memory buffer size

**Network Efficiency:**
- Forward chunks immediately (no batching delay)
- Use efficient SSE formatting
- Compress if needed (future enhancement)

**CPU Usage:**
- Stream parsing should be lightweight
- Avoid heavy processing per chunk
- Use efficient JSON parsing

### 6.2 Frontend Performance

**Rendering Optimization:**
- Debounce markdown parsing (100ms)
- Use `useMemo` for parsed markdown
- Avoid re-rendering on every chunk
- Batch state updates if chunks arrive rapidly

**Memory Management:**
- Accumulated text grows with summary length
- Monitor memory usage for very long summaries
- Consider pagination for extremely long content

**Network Efficiency:**
- SSE is efficient for one-way streaming
- No additional network overhead
- Connection pooling handled by browser

### 6.3 Scalability

**Concurrent Streams:**
- Each job has independent stream
- No shared state between streams
- Backend can handle multiple concurrent streams
- Monitor resource usage with many concurrent jobs

**Rate Limiting:**
- Existing rate limiting applies
- No additional rate limits needed for streaming
- Consider throttling chunk forwarding if needed

## 6.4 Animation Guidelines & Best Practices

### 6.4.1 Animation Principles

- Use animations to provide feedback, not as decoration
- Keep animation durations between 200-400ms for responsiveness

## 15. Appendix

### 15.1 DashScope Stream Format Reference

**Example Stream:**
```
data: {"output":{"choices":[{"message":{"content":"Hello"}}]},"request_id":"abc123"}

data: {"output":{"choices":[{"message":{"content":" world"}}]},"request_id":"abc123"}

data: {"output":{"choices":[{"message":{"content":"!"}}]},"request_id":"abc123"}

data: [DONE]
```

**Content Extraction:**
- Path: `output.choices[0].message.content`
- May be incremental delta or full content
- Accumulate deltas to build full response

### 15.2 SSE Event Format Reference

**Standard SSE Format:**
```
event: progress
data: {"status":"generating","chunk":"text","progress":87}

```

**Multiple Events:**
- Use `event:` field to distinguish event types
- `data:` field contains JSON payload
- Empty line separates events

### 15.3 Code Examples

**Backend Stream Parsing (Pseudocode):**
```typescript
async function parseDashScopeStream(stream, onChunk) {
  const readline = require('readline');
  const rl = readline.createInterface({ input: stream });
  let accumulatedContent = '';
  
  for await (const line of rl) {
    if (line.startsWith('data: ')) {
      const jsonStr = line.slice(6);
      if (jsonStr === '[DONE]') break;
      
      const data = JSON.parse(jsonStr);
      const chunk = data.output?.choices?.[0]?.message?.content || '';
      
      if (chunk) {
        accumulatedContent += chunk;
        onChunk(chunk);
      }
    }
  }
  
  return { content: accumulatedContent };
}
```

**Frontend Chunk Accumulation (Pseudocode):**
```typescript
const [streamedText, setStreamedText] = useState('');

eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  
  if (data.status === 'generating' && data.chunk) {
    setStreamedText(prev => prev + data.chunk);
  }
};
```
